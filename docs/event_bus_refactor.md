# Event Bus Refactor: Services Emit, TUI/Adapters Subscribe

## Status

Draft. Awaiting approval.

## Goal

Replace the periodic `PollTickMsg` (fires every 2s, full workspace reload) with
event-driven, per-item targeted refreshes. State-transition events originate from
the service layer. The `event.Bus` is a shared singleton in the application
composition layer, used by services (as emitters), the TUI (as a subscriber),
and adapters (as subscribers).

## Architecture

```
┌─────────────────────────────────────────────────────────────────────┐
│  Composition Layer                                                    │
│                                                                      │
│  event.Bus ◀──subscribe── TUI (DomainEventMsg → App.Update)          │
│      │                    ├──subscribe── Adapters                     │
│      │                                                                 │
│      └──▶ publish ── Services (emit on state transition)              │
│                            SessionService                            │
│                            TaskService                                │
│                            PlanService                                │
└─────────────────────────────────────────────────────────────────────┘

TUI ──calls──▶ Orchestrator (combines services into workflow actions)
TUI ◀──result messages── Orchestrator

Orchestrator: pure publisher of higher-level workflow events.
             Never subscribes to the bus.
             Owns: Foreman, ReviewPipeline, Resumption, ImplementationService
             (as workflow coordinators, not service lifecycle owners)
```

**Key architectural rules:**

1. Services own state and emit events on every transition. They are the source of truth.
2. Orchestrator coordinates services but does not hold the bus for service-layer events.
3. Orchestrator never subscribes to the bus — it only publishes.
4. TUI subscribes to the bus and bridges events to its update loop.
5. Adapters subscribe to the bus — already implemented, unchanged.
6. `PollTickMsg` is eliminated for work item and task state.

## What Orchestrator Does with the Bus

The orchestrator **never reads from the bus**. It only publishes.

After this refactor, orchestrator publishes are:

| Publisher | Event | Keep/Remove |
|---|---|---|
| `Foreman` | `EventAgentQuestionRaised` | Keep — owned by Foreman |
| `QuestionRouter` | `EventAgentQuestionAnswered` | Keep — owned by QuestionRouter |
| `ReviewPipeline` | `EventReviewStarted/Completed/CritiquesFound/ReimplementationStarted` | Keep — owned by ReviewPipeline |
| `Resumption` | `EventAgentSessionResumed` | Keep — owned by Resumption |
| `ImplementationService` | `EventWorktreeCreated/Reused` | Keep — repo lifecycle, not service transitions |
| `ImplementationService` | `EventAgentSessionStarted/Completed/Failed/Interrupted` | **Remove** — service emits |
| `ImplementationService` | `EventWorkItemCompleted` | **Remove** — service emits |
| `PlanningService` | `EventWorkItemPlanning` | **Remove** — service emits |
| `PlanningService` | `EventPlanGenerated/Failed` | Keep — generated by orchestrator |
| `PlanService` | `EventPlanApproved/Rejected/SubmittedForReview/Revised` | **Add** — move from orchestrator/TUI |

## Service Lifecycle Management

### Current Problem

`SettingsService.rebuildServices()` (lines 443–688) does two unrelated things:

1. Manages config snapshot and workspace settings (its actual job)
2. Creates and wires all services, orchestrators, harnesses, adapters, and the event bus

The settings view knows how to build the entire service graph. This is a separation of
concerns violation.

### Desired Design

A dedicated `ServiceManager` owns the service graph lifecycle. `SettingsService`
only manages config, workspace selection, and credentials. When config changes, it
notifies `ServiceManager` to rebuild.

```
SettingsService            → manages config, workspace, credentials
                            notifies ServiceManager on changes

ServiceManager             → owns service + orchestrator + harness + bus lifecycle
                            builds and tears down the service graph
                            owns the bus (shared singleton)

SettingsView / TUI         → calls SettingsService to change config
                            receives Services via ServiceManager.GetServices()
                            subscribes to ServiceManager for updates
```

**`ServiceManager` interface:**

```go
// ServiceManager owns the complete service graph lifecycle.
// It is the single place responsible for building and rebuilding services.
type ServiceManager struct {
    transacter  atomic.Transacter[repository.Resources]
    eventRepo   repository.EventRepository
    secretStore config.SecretStore
    cfg         *config.Config
    mu          sync.RWMutex
    services    *Services // nil until Init() or after first Rebuild()
}

// Init builds the initial service graph.
func (sm *ServiceManager) Init(ctx context.Context, cfg *config.Config) error {
    sm.mu.Lock()
    defer sm.mu.Unlock()
    svcs, err := sm.buildServices(ctx, cfg)
    if err != nil { return err }
    sm.services = svcs
    sm.cfg = cfg
    return nil
}

// Rebuild tears down the current graph and builds a new one.
// Called by SettingsService when config changes.
func (sm *ServiceManager) Rebuild(ctx context.Context, cfg *config.Config) error {
    sm.mu.Lock()
    defer sm.mu.Unlock()

    // Tear down: close bus (closes all adapter subscription goroutines)
    if sm.services != nil && sm.services.Bus != nil {
        sm.services.Bus.Close()
    }

    svcs, err := sm.buildServices(ctx, cfg)
    if err != nil { return err }
    sm.services = svcs
    sm.cfg = cfg
    return nil
}

// GetServices returns the current service graph.
// Safe to call concurrently.
func (sm *ServiceManager) GetServices() *Services {
    sm.mu.RLock()
    defer sm.mu.RUnlock()
    return sm.services
}

// Close shuts down the service graph.
func (sm *ServiceManager) Close() {
    sm.mu.Lock()
    defer sm.mu.Unlock()
    if sm.services != nil && sm.services.Bus != nil {
        sm.services.Bus.Close()
    }
}
```

**`Transition()` — emit after commit:**

```go
func (s *SessionService) Transition(ctx context.Context, id string, to domain.SessionState) error {
    var committed stateChangeEvent
    err := s.transacter.Transact(ctx, func(ctx context.Context, res repository.Resources) error {
        item, err := res.Sessions.Get(ctx, id)
        if err != nil { return newNotFoundError("work item", id) }

        from := item.State
        if !canTransition(from, to) {
            return newInvalidTransitionError(
                workItemStateName(from), workItemStateName(to), "work item",
            )
        }

        item.State = to
        item.UpdatedAt = time.Now()
        if err := res.Sessions.Update(ctx, item); err != nil { return err }

        // Capture after DB write — will emit only if transaction commits.
        committed = stateChangeEvent{from: from, to: to, item: item}
        return nil
    })
    if err != nil { return err }

    // Transaction committed. Emit outside the transaction.
    if committed.item.ID != "" {
        s.emitStateChange(context.Background(), committed.from, committed.to, committed.item)
    }
    return nil
}

type stateChangeEvent struct {
    from, to domain.SessionState
    item      domain.Session
}
```

**`emitStateChange()` — fires asynchronously:**

```go
func (s *SessionService) emitStateChange(ctx context.Context, from, to domain.SessionState, item domain.Session) {
    if s.bus == nil { return }

    evt := domain.SystemEvent{
        ID:          domain.NewID(),
        EventType:   s.stateToEventType(to),
        WorkspaceID: item.WorkspaceID,
        Payload:     fmt.Sprintf(`{"work_item_id":%q,"state":%q}`, item.ID, to),
        CreatedAt:   time.Now().UTC(),
    }
    go func() {
        if err := s.bus.Publish(ctx, evt); err != nil {
            slog.Warn("failed to publish work item state event", "error", err, "work_item_id", item.ID)
        }
    }()
}

func (s *SessionService) stateToEventType(state domain.SessionState) string {
    switch state {
    case domain.SessionPlanning:    return string(domain.EventWorkItemPlanning)
    case domain.SessionPlanReview: return string(domain.EventWorkItemPlanReview)
    case domain.SessionApproved:   return string(domain.EventWorkItemApproved)
    case domain.SessionImplementing: return string(domain.EventWorkItemImplementing)
    case domain.SessionReviewing: return string(domain.EventWorkItemReviewing)
    case domain.SessionCompleted: return string(domain.EventWorkItemCompleted)
    case domain.SessionFailed:   return string(domain.EventWorkItemFailed)
    case domain.SessionMerged:   return string(domain.EventWorkItemMerged)
    default:                     return ""
    }
}
```

### Phase III.C — `DomainEventMsg` Handler

         domain.EventAgentSessionInterrupted,
         domain.EventAgentSessionResumed:

        }

    // Plan lifecycle → reload plan
    case domain.EventPlanGenerated,
         domain.EventPlanApproved,
         domain.EventPlanRejected,
         domain.EventPlanRevised,
         domain.EventPlanSubmittedForReview:
        workItemID := extractWorkItemID(msg.Event.Payload)
        if workItemID != "" {
            cmds = append(cmds, LoadPlanForSessionCmd(a.svcs.Plan, workItemID))
        }

    // Review events → reload tasks
    case domain.EventReviewStarted,
         domain.EventReviewCompleted,
         domain.EventCritiquesFound:
        workItemID := extractWorkItemID(msg.Event.Payload)
        if workItemID != "" {
            cmds = append(cmds, LoadTasksForSessionCmd(a.svcs.Task, workItemID))
        }

    // Question events → reload questions
    case domain.EventAgentQuestionRaised,
         domain.EventAgentQuestionAnswered:
        sessionID := extractSessionID(msg.Event.Payload)
        if sessionID != "" {
            cmds = append(cmds, LoadQuestionsCmd(a.svcs.Question, sessionID))
        }

    // Higher-level events that don't need targeted reload (already handled by completion messages)
    case domain.EventImplementationStarted,
         domain.EventReimplementationStarted,
         domain.EventPRMerged,
         domain.EventPRReviewStateChanged:
        // No-op: these don't change the work item state directly
    }

    // Re-render if viewing the affected work item
    if a.currentWorkItemID != "" {
        cmds = append(cmds, a.updateContentFromState())
    }

    return a, tea.Batch(cmds...)

## Eliminate PollTickMsg

### Remove from `Init()`

```go
// Before (in App.Init()):
cmds = append(cmds,
    LoadSessionsCmd(a.svcs.Session, a.svcs.WorkspaceID),
    LoadTasksCmd(a.svcs.Task, a.svcs.WorkspaceID),
    PollTickCmd(),           // ← REMOVE
    // ...
)

// After:
cmds = append(cmds,
    LoadSessionsCmd(a.svcs.Session, a.svcs.WorkspaceID),
    LoadTasksCmd(a.svcs.Task, a.svcs.WorkspaceID),
    // PollTickCmd removed — TUI now event-driven via DomainEventMsg
)
```

### Keep `PollTickMsg` for Toast Pruning

```go
case PollTickMsg:
    a.toasts.Prune()
    // Heartbeat is separate; consider moving to event-driven later.
    if a.svcs.InstanceID != "" {
        cmds = append(cmds, HeartbeatCmd(a.svcs.Instance, a.svcs.InstanceID))
    }
    // No LoadSessionsCmd / LoadTasksCmd — removed.
    cmds = append(cmds, PollTickCmd()) // still fires for toast pruning
    return a, tea.Batch(cmds...)
```

**Note:** `PollTickCmd()` still fires so toasts are pruned. This is acceptable.
The critical thing is that work item/task state is no longer poll-driven.

### Cascade: Verify All Paths Are Event-Driven

With the poll gone, every code path that changes state must result in a reload.
The completion message handlers (`ImplementationCompleteMsg`, etc.) still dispatch
targeted load commands. Dual sourcing (both message handler AND `DomainEventMsg`)
is harmless — `SessionsLoadedMsg` does an upsert, so a second call with the same
data is a no-op.

## Event Payloads: Consistent Format

Currently event payloads are inconsistent:

- `PlanningService.emitPlanningStartedEvent`: `{"work_item_id":"...","session_id":"..."}` (manual string)
- `ImplementationService.newSessionEventPayload`: Go struct marshaled to JSON
- `cmds.go emitPlanApproved`: `map[string]any` marshaled to JSON

**Decision:** All event payloads use `map[string]any` with consistent lowercase keys,
marshaled with `encoding/json`. Helper functions in `internal/event/payload.go`:

```go
package event

import "encoding/json"

func WorkItemStatePayload(workItemID, workspaceID, state string) string {
    m := map[string]any{
        "work_item_id": workItemID,
        "workspace_id": workspaceID,
        "state":        state,
    }
    b, _ := json.Marshal(m)
    return string(b)
}

func SessionPayload(session domain.Task) string {
    m := map[string]any{
        "session_id":    session.ID,
        "work_item_id":  session.WorkItemID,
        "workspace_id":  session.WorkspaceID,
        "phase":         string(session.Phase),
    }
    b, _ := json.Marshal(m)
    return string(b)
}
```

All emitters (services and remaining orchestrator events) use these helpers.
The TUI's `extractWorkItemID()` and `extractSessionID()` rely on lowercase `work_item_id`
and `session_id` keys.

## Open Questions

1. **`emitWorkItemCompleted` payload split:** The orchestrator's `emitWorkItemCompleted`
   includes `branch`, `review`, `sub_plan_content`. The service emits a basic
   `EventWorkItemCompleted`. Should the richer payload stay in the orchestrator,
   or should the service fetch it? Recommendation: keep in orchestrator. The service
   doesn't have access to `branch`/`review`/`sub_plan_content`. Splitting the event
   is more complex than it's worth.

2. **`TaskService.Interrupt()` as a service method:** Currently the orchestrator
   calls `TaskService.Fail()` with a special exit code to represent interrupt.
   Should there be a dedicated `Interrupt()` method on the service, or should the
   orchestrator call `Fail()`? Recommendation: add `Interrupt()` to the service
   so the service emits `EventAgentSessionInterrupted` directly. This keeps the
   service as the single source of truth for session state.

3. **Live instance heartbeat:** Still uses `PollTickMsg` → `HeartbeatCmd`.
   Could be event-driven (instances emit heartbeat events). Low priority — leave
   the poll for now.

4. **`EventWorktreeCreated` / `EventWorktreeReused`:** Currently emitted by
   `ImplementationService` but not tied to a service transition. These are
   fine to stay in the orchestrator — they're workflow events, not state
   transitions.

5. **Test bus in tests:** All test files that construct services need to be
   updated to pass `nil` or a test `*event.Bus`. A shared test helper
   `testBus := event.NewBus(event.BusConfig{})` in each test file is sufficient.
